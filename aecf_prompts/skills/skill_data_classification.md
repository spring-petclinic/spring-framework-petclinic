## Skill ID
`aecf_data_classification`

## Layer
`Explorators` — Knowledge Extraction Skills

## Description
Explore and discover the project's data models, schemas, migrations, and DTOs. It applies deterministic data classification heuristics by sensitivity and generates a governance proposal structured in YAML. Feed audit, enforcement and fix skills with an audited and classified data inventory.

## Objective
Generate a complete and classified inventory of all project data artifacts in sufficient detail for downstream skills (audit, enforcement, fix) to operate without the need for manual rediscovery.

## Inputs

### Mandatory:
- **Scope**: module, service, folder or feature to explore

### Optional:
- **TOPIC**: analysis identifier
- Custom classification taxonomy (if it does not exist, use defaults: PUBLIC / INTERNAL / CONFIDENTIAL / SENSITIVE / PII)
- Reference governance scheme: `aecf_prompts/_governance/DATA_GOVERNANCE_SPEC_SCHEMA.md`

------------------------------------------------------------

## Steps

### Step 1: Data Artifact Discovery
Scan the defined scope and locate all data artifacts:
- ORM models / Entity definitions
- Database migration files
- DTO / Data Transfer Objects
- API schemas (OpenAPI, GraphQL, Protobuf)
- Configuration files with data bindings
- Seed data / fixture files

Output: Inventory list of all discovered artifacts with path and type.

### Step 2: Field-Level Extraction
For each discovered artifact, deterministically extract:

| Attribute | Description |
|-----------|-------------|
| `table_name` | Name of the table / collection / entity |
| `field_name` | Name of each field / column / property |
| `field_type` | Data type (string, int, datetime, uuid, etc.) |
| `nullable` | Whether the field accepts null values |
| `is_pk` | Whether the field is a primary key |
| `is_fk` | Whether the field is a foreign key (and target) |
| `relationships` | One-to-one / one-to-many / many-to-many references |
| `comments` | Inline code comments or docstrings on the field |
| `constraints` | Unique, index, length, regex constraints |
| `default_value` | Default value if defined |

### Step 3: Deterministic Classification Heuristics
Apply classification rules in strict priority order (first match wins):

**Priority 1 — Explicit annotation**: If field has a governance annotation (e.g., `@sensitive`, `@pii`, `@classification`) → use declared value.

**Priority 2 — Field name pattern matching** (case-insensitive):

| Pattern | Classification |
|---------|---------------|
| `*password*`, `*secret*`, `*token*`, `*api_key*`, `*private_key*` | SENSITIVE |
| `*ssn*`, `*social_security*`, `*passport*`, `*driver_license*` | PII |
| `*email*`, `*phone*`, `*mobile*`, `*address*`, `*birth*`, `*dob*`, `*gender*` | PII |
| `*name*`, `*first_name*`, `*last_name*`, `*full_name*` | PII |
| `*ip_address*`, `*device_id*`, `*cookie*`, `*fingerprint*` | PII |
| `*salary*`, `*income*`, `*credit_*`, `*bank_*`, `*iban*`, `*card_number*` | SENSITIVE |
| `*health*`, `*diagnosis*`, `*medication*`, `*medical_*` | SENSITIVE |
| `*internal_*`, `*employee_*`, `*staff_*` | CONFIDENTIAL |
| `*id*` (when FK), `*uuid*`, `*reference_*` | INTERNAL |
| `*created_at*`, `*updated_at*`, `*deleted_at*` | INTERNAL |

**Priority 3 — Relationship inference**: If a table contains any PII or SENSITIVE field, all FK fields pointing to it inherit CONFIDENTIAL classification.

**Priority 4 — Default**: PUBLIC

**Confidence scoring**: Each classification must include a confidence score (HIGH / MEDIUM / LOW) based on how it was determined:
- Priority 1 → HIGH
- Priority 2 → HIGH (exact match) or MEDIUM (partial match)
- Priority 3 → MEDIUM
- Priority 4 → LOW

### Step 4: Relationship Graph Construction
For each entity, document:
- Direct relationships (FK constraints)
- Implied relationships (naming conventions: `*_id` → FK pattern)
- Cardinality: ONE-TO-ONE / ONE-TO-MANY / MANY-TO-MANY

### Step 5: Governance Proposal Generation
Generate a structured governance proposal covering:
- Classification summary by sensitivity level
- High-risk fields requiring immediate attention
- Missing governance annotations (fields that should be annotated but aren't)
- Retention policy recommendations per classification
- Access control recommendations per classification
- Data minimization opportunities (fields that may be over-collected)

### Step 6: Deterministic YAML Output Generation
Generate `data_classification.yaml` as part of the output document.

YAML structure:

```yaml
# AECF Explorator — Data Classification Output
# Skill: aecf_data_classification
# Generated: {{TIMESTAMP}}
# Scope: {{SCOPE}}
# Topic: {{TOPIC}}

metadata:
  skill_version: "1.0"
  generated_by: "aecf_data_classification"  # Generator skill ID for this artifact, not execution user_id
  schema_reference: "aecf_prompts/_governance/DATA_GOVERNANCE_SPEC_SCHEMA.md"
  deterministic: true

entities:
  - entity_name: "<table_or_entity_name>"
    source_file: "<relative_path_to_artifact>"
    artifact_type: "<orm_model|migration|dto|schema|fixture>"
    overall_classification: "<PUBLIC|INTERNAL|CONFIDENTIAL|SENSITIVE|PII>"
    fields:
      - name: "<field_name>"
        type: "<data_type>"
        classification: "<PUBLIC|INTERNAL|CONFIDENTIAL|SENSITIVE|PII>"
        confidence: "<HIGH|MEDIUM|LOW>"
        classification_reason: "<priority_N_rule: description>"
        is_pk: <true|false>
        is_fk: <true|false>
        fk_target: "<entity.field or null>"
        nullable: <true|false>
        constraints: []
        governance_note: "<optional note>"

relationships:
  - from_entity: "<entity_name>"
    from_field: "<field_name>"
    to_entity: "<entity_name>"
    to_field: "<field_name>"
    cardinality: "<ONE-TO-ONE|ONE-TO-MANY|MANY-TO-MANY>"
    classification_impact: "<inherited CONFIDENTIAL from PII parent? YES/NO>"

governance_summary:
  total_entities: <int>
  total_fields: <int>
  classification_breakdown:
    PUBLIC: <int>
    INTERNAL: <int>
    CONFIDENTIAL: <int>
    SENSITIVE: <int>
    PII: <int>
  high_risk_fields: <int>
  missing_annotations: <int>
  data_minimization_candidates: <int>
  retention_policy_gaps: <int>
  overall_risk_level: "<LOW|MEDIUM|HIGH|CRITICAL>"
```
