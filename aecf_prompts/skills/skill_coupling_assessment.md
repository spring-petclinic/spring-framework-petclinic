## Skill ID
`aecf_coupling_assessment`

## Description
Focused architectural audit that detects tight coupling, unstable module boundaries, cyclic dependencies, shared-state leaks, and other coupling hotspots that make refactoring risky or expensive.

## When to Use
- Before `aecf_refactor` when you suspect coupling is slowing safe change
- After `aecf_codebase_intelligence` or `aecf_tech_debt_assessment` when architectural hotspots are visible
- Before splitting modules, extracting services, or introducing clearer boundaries
- During due diligence on legacy code where dependency chains are hard to reason about

## When NOT to Use
- You already need to implement the refactor immediately and the problematic files are fully known -> use `aecf_refactor`
- You only need standards/style feedback -> use `aecf_code_standards_audit`
- You need a broader debt inventory instead of a coupling-focused audit -> use `aecf_tech_debt_assessment`

---

## Coupling Finding Types

The report MUST classify findings into one or more of these types:

- **Layer Leakage**: presentation, orchestration, persistence, or infrastructure concerns cross boundaries improperly
- **Cyclic Dependency**: two or more modules require each other directly or indirectly
- **Shared Mutable State**: modules coordinate through hidden state instead of explicit contracts
- **Wide Fan-In / Fan-Out**: a module becomes a central hotspot with too many upstream or downstream dependencies
- **Temporal Coupling**: operations only work when invoked in a fragile order that is not captured by an explicit contract
- **Testability Coupling**: production units cannot be isolated without broad fixture or environment setup

## Severity Heuristics

Use deterministic severity rules based on repository evidence:

| Severity | Deterministic trigger |
|----------|-----------------------|
| **HIGH** | Cycles across architectural layers, shared mutable state across modules, or hotspots that block isolated testing/refactoring |
| **MEDIUM** | Excessive fan-in/fan-out, hidden dependency chains, or boundary leaks with contained blast radius |
| **LOW** | Local coupling smells that are still easy to isolate and do not currently block testing or change |

---

## Execution Steps

### Step 1: Dependency Surface Discovery
Locate imports, injected services, shared config/state access, message flows, and cross-module calls in scope.

### Step 2: Boundary Mapping
Map the expected architectural boundaries and contrast them with the actual dependency graph found in the repository.

### Step 3: Coupling Hotspot Classification
Classify each hotspot by finding type, severity, impacted files, dependency direction, and architectural consequence.

### Step 4: Refactor Readiness Analysis
For every MEDIUM or HIGH finding, document the smallest safe seam for change and the expected risk if refactoring proceeds without prior isolation.

### Step 5: Test Recommendation Mapping
For every MEDIUM or HIGH finding, specify the characterization or regression tests required to freeze current behavior before change. These recommendations MUST map to `aecf_new_test_set`.

### Step 6: Remediation Routing
For every actionable finding, recommend the downstream AECF skill chain using only existing skill IDs.
