## Skill ID
`aecf_productivity`

## Description
Analyzes AECF delivery activity for one person, several people, or an entire group using `TOPICS_INVENTORY` and `CHANGELOG` evidence to produce productivity KPIs and an executive summary.

## Objective
Provide a governance-safe productivity view based on verified AECF execution evidence, highlighting throughput, topic cadence, file touch coverage, and collaboration patterns without relying on raw commit counts alone.

## Supported Request Shapes

- Single person: `user_id=<id>`
- Multiple people: `user_ids=<id1,id2,...>`
- Full population: `all_users=true` or prompt intent equivalent
- Optional date window: `date_from=<YYYY-MM-DD>`, `date_to=<YYYY-MM-DD>`
- Optional topic filter: exact topics, topic prefixes, or explicit list
- Optional file filter: file paths, path prefixes, or repository areas to backtrack touched topics

## Minimum KPI Set

The report MUST calculate, when evidence exists:

1. `active_days`
2. `topics_touched`
3. `topics_closed`
4. `topics_per_active_day`
5. `changelog_entries`
6. `artifacts_generated`
7. `files_touched`
8. `files_per_topic`
9. `topic_closure_rate`
10. `date_window_coverage`

For group analysis the report MUST also include:

1. Per-member KPI table
2. Group totals
3. Group averages
4. Distribution commentary (high/low dispersion, concentration, imbalance)
5. Shared topic overlap or specialization pattern when evidence supports it

## Mandatory Interpretation Rules

- Productivity MUST be grounded in verified AECF artifacts, not in generic assumptions.
- A high number of touched files without topic closure MUST be interpreted as flow pressure, not automatically as high performance.
- A high closure rate with narrow file spread MUST be interpreted as focused delivery, not automatically as low scope.
- Group output MUST first show each member, then the aggregate group view.
- Any KPI omitted due to missing evidence MUST be listed under `Coverage Gaps`.

## Execution Steps

### Step 1: Resolve Analysis Scope
Resolve the population, date window, topic constraints, and file constraints. If no date window is provided, use the verified evidence span and report it explicitly.

### Step 2: Load Productivity Evidence
Load the relevant `TOPICS_INVENTORY` and `CHANGELOG` files. Normalize the evidence into a comparable table keyed by user, topic, timestamp, and available file references.

### Step 3: Apply Deterministic Filters
Apply the exact requested filters in this order: population, date range, topics, files. Every excluded record MUST be excluded for a documented reason.

### Step 4: Build Person-Level KPIs
Compute the minimum KPI set for each requested user. If the request is for a single person, this section remains mandatory.

### Step 5: Build Group-Level KPIs
When more than one user is in scope, aggregate totals, averages, dispersion, and overlap. The group result MUST never replace the per-person breakdown.

### Step 6: Executive Interpretation
Produce an executive summary describing cadence, delivery balance, concentration by topic or file area, and notable coverage gaps.

### Step 7: Recommendations
Recommend only governance-safe next actions, such as balancing load, closing stale topics, or improving traceability when evidence is incomplete.
