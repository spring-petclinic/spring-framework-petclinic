## Skill ID
`aecf_application_lifecycle`

## Description
Generate an application development lifecycle document aligned to one accepted project-management methodology and explicitly map which AECF skills should be used in each phase.

## When to Use
- Define a project operating model before starting delivery
- Standardize how AECF skills fit inside a chosen governance framework
- Align architecture, delivery, audit, release, and improvement activities under one lifecycle reference
- Create a methodology-specific lifecycle guide for onboarding or governance review

## When NOT to Use
- You need to implement code changes -> use `aecf_new_feature`, `aecf_refactor`, or `aecf_hotfix`
- You need to document the existing codebase itself -> use `aecf_document_legacy`
- You need a release decision for a current increment -> use `aecf_release_readiness`

## ACCEPTED METHODOLOGY CATALOG

The exact accepted values are:

| Identifier | Description |
| --- | --- |
| `AGILE` | Generic agile lifecycle umbrella when the user wants iterative delivery without committing to Scrum or Kanban specifically |
| `PMBOK` | PMI process-group based lifecycle |
| `PRINCE2` | Stage-gated lifecycle with management products and control boundaries |
| `SCRUM` | Sprint-based iterative lifecycle |
| `KANBAN` | Continuous-flow lifecycle |
| `ISO_12207` | ISO/IEC/IEEE software lifecycle processes |
| `HYBRID_PMBOK_SCRUM` | Governance-heavy hybrid using PMBOK controls plus Scrum delivery |

If the input does not resolve to one of these values, execution MUST stop in `INTAKE`.
