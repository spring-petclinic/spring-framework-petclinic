## EXECUTION MANDATE (IMPERATIVE)

When this skill is invoked, the AI MUST:

1. **PARSE** the prompt for all required project parameters (project name, type, language/framework, database)
2. **VALIDATE** every required parameter against the supported catalogs defined in this skill
3. **BLOCK AND ASK** if ANY required parameter is missing, ambiguous, or not in the support catalog — create `00_NEW_PROJECT_INTAKE.md` and STOP
4. **PLAN** the operational skeleton once all parameters are confirmed, explicitly documenting the chosen base approach and constraints
5. **AUDIT_PLAN** review that plan before any scaffold or file materialization begins
6. **SCAFFOLD** the complete project structure only after the intake is complete and the plan has passed audit
7. **GENERATE** README.md, config files, source stubs, and AECF_PROJECT_CONTEXT.md
8. **IMPLEMENT** materialize all project files physically in the workspace using `<AECF_FILE_CHANGES>` blocks
9. **AUDIT_IMPLEMENT** validate that the scaffold was really materialized under the client workspace and that critical files are coherent for the chosen stack
10. **FIX_CODE** only when `AUDIT_IMPLEMENT = NO-GO`, then re-run `AUDIT_IMPLEMENT` until `GO`

**MANDATORY PHASE GATES (NO SKIP)**:
- **NO SCAFFOLD BEFORE INTAKE COMPLETE**: all 4 mandatory inputs (project_name, project_type, language_framework, database) MUST be resolved before any file creation begins
- **NO SCAFFOLD BEFORE PLAN GO**: the operational skeleton must be planned and pass `AUDIT_PLAN` before scaffolding starts
- **BLOCK ON AMBIGUITY**: if 2+ language/framework options could match the prompt → STOP, ask to confirm exactly one
- **BLOCK ON MISSING DB**: if database=YES but no db type is specified → STOP, ask which one from the catalog
- **BLOCK ON UNKNOWN TYPE**: if project_type is not in the supported catalog or cannot be resolved as `domain_aligned_project` through available knowledge packs / semantic profiles → STOP, show the catalog and ask

**MANDATORY POST-EXECUTION GOVERNANCE (per SKILL_DISPATCHER)**:
- **UPDATE** `aecf_prompts/<DOCS_ROOT>/<user_id>/AECF_TOPICS_INVENTORY.json` for TOPIC lifecycle and **REGENERATE** `aecf_prompts/<DOCS_ROOT>/<user_id>/AECF_TOPICS_INVENTORY.md` (Step 4.1)
- **APPEND** one execution entry to `aecf_prompts/<DOCS_ROOT>/<user_id>/AECF_CHANGELOG.md` (Step 4.2)

**FORBIDDEN**:
- ❌ Responding only in chat without creating files
- ❌ Creating ANY project file before all mandatory parameters are confirmed
- ❌ Creating a generic/blank scaffold when a concrete type-aware scaffold exists
- ❌ Assuming the database is "none" if the user has not explicitly stated it
- ❌ Bundling multiple unresolved mandatory parameters into the same clarification turn when one blocking question is enough to continue deterministically

## Skill ID
`aecf_new_project`

## Description
Bootstrap a complete, production-ready project skeleton from scratch.
Collects project type, language/framework, and database requirements,
validates them against a concrete catalog, and generates the full folder structure,
README, config files, source stubs, tests scaffold, CI/CD workflow, and an
AECF_PROJECT_CONTEXT.md file so the project is immediately usable.

## When to Use
- Starting a new project from scratch
- Bootstrapping a standardized project skeleton for a known type/stack
- Team onboarding to a new project with consistent structure
- Creating a new service/library within a multi-repo environment

## When NOT to Use
- Adding features to an existing project → use `aecf_new_feature`
- Generating context for an existing project → use `aecf_project_context_generator`
- Scaffolding a single module within an existing project → use `aecf_new_feature`

- A bare domain ID such as `python` or `cobol` is not enough by itself for `domain_aligned_project`; the user must choose one or more semantic profiles so the final stack becomes deterministic.

- If the user enters a profile that does not exist yet, create it under `.aecf/knowledge/<domain>/semantic_profiles/` before continuing.
- Workspace-local knowledge under `.aecf/knowledge/` extends the bootstrap catalog; it does not replace homonymous component assets.
- The generated skeleton must follow the loaded domain pack and semantic profiles, preserving contracts and conventions of that domain.
- Use this mode for COBOL, legacy platforms, proprietary domains, and user-augmented semantic profiles.

## PHASE 1 — INTAKE

### Goal
Collect and validate ALL required parameters. If any are missing or ambiguous,
generate `00_NEW_PROJECT_INTAKE.md` and STOP.

### Sequential Intake Rule

- Ask exactly ONE unresolved mandatory parameter per INTAKE turn.
- Priority order for blocking questions: `project_name` -> `project_type` -> `language_framework` -> `database`.
- Do not ask for the next parameter until the current blocking parameter has been explicitly resolved by the user.
- For every blocking question, provide exactly 3 concrete options the user can choose from.
- One of those 3 options MUST be a recommended default answer and must be clearly marked as recommended.
- If the blocking parameter is `project_name`, ask for `project_name` directly and provide 3 valid candidate names derived from the prompt.
- If the blocking parameter is `database`, ask for `database` directly and provide 3 valid database choices aligned with the requested stack, including `none` only when stateless execution is plausible.
- When the machine-usable answer differs from the visible label, encode the option as `<visible label> => <machine_usable_value>`.
- Recommended options MUST be marked inline as `[recommended] <visible label> => <machine_usable_value>` so the runtime can surface them correctly.

### Validation Rules

1. **project_name**:
   - Required. No default possible.
   - Must be a valid directory name: `[a-z0-9_-]`, no spaces, max 64 chars.
   - If it conflicts with an existing directory in workspace → WARN, include in INTAKE.

2. **project_type**:
  - Required. Must exactly match one of the catalog IDs above.
   - If the user describes a type in natural language, resolve to catalog ID.
  - If the stack resolves through available knowledge packs or semantic profiles but not through the static archetypes, normalize to `domain_aligned_project`.
   - If ambiguous between 2+ types → BLOCK (list options, ask to pick one).

3. **language_framework**:
  - Required. Must match one of the sub-catalog IDs for the resolved `project_type`.
  - For `domain_aligned_project`, it must resolve to a semantic profile ID or explicit composed stack identifier from component/workspace knowledge.
  - If the user only provides a domain ID, block and ask which semantic profiles to use.
  - Allow multi-select semantic profiles and allow new workspace profiles under `.aecf/knowledge/<domain>/semantic_profiles/` when the requested profile does not exist yet.
   - If ambiguous between 2+ options → BLOCK (list options, ask to pick one).
   - Never guess the default language if the user has not indicated preference.

4. **database**:
   - Required. No default. The AI MUST NOT assume `none`.
   - If the user says "yes, I want a database" without specifying which → BLOCK.
   - If the user says "no database" or "stateless" → resolve to `none`.

## PHASE 6 — AUDIT_IMPLEMENT

### Goal
Audit the scaffold that was physically materialized in the client workspace.
This phase validates structure completeness, stack consistency, and file content
quality before the skill is considered complete.

### Mandatory Behavior

1. **Audit the real scaffold under `output_path`** — do not audit only the narrative artifacts.

3. **Validate stack coherence**: the generated files, dependencies, configs, and stubs must match the resolved `project_type`, `language_framework`, and `database`.
4. **Detect placeholder-only or empty files** in critical project locations and treat them as findings.
5. **Optional runtime verification**: if the scaffold includes immediately runnable smoke checks and evidence exists, mention it. If no runtime verification was executed yet, audit structural completeness instead of forcing test evidence.
6. **Return `NO-GO`** when critical scaffold elements are missing, wrongly located, empty, inconsistent with the selected stack, or clearly unusable as a project bootstrap.
7. **Return `GO`** only when the scaffold is physically present, coherent, and complete enough for a developer to start from it.

- `## Scaffold Presence Checks`
