`aecf_explain_behavior`

## Description
Explains why the system currently behaves as observed, using repository evidence discovered via IDE LLM and governed by AECF quality gates.

## Responsibility Boundary (Mandatory)

IDE LLM responsibilities:
- Search and inspect repository artifacts.
- Locate candidate files, functions, configs, tests, and logs.
- Gather verifiable evidence and references.

AECF responsibilities:
- Validate completeness of WORKING_CONTEXT.
- Build behavioral causality and technical explanation.
- Apply governance criteria and risk classification.
- Emit structured, auditable output with GO/NO-GO decision.

## Execution Context Isolation (Mandatory)

WORKING_CONTEXT rules:
- WORKING_CONTEXT MUST be generated and consumed inside the same active execution context.
- WORKING_CONTEXT MUST NOT be written to or read from a shared static file.
- If persistence is required for traceability/debug, it MUST use a unique execution identifier (`EXECUTION_ID`).
- Persisted path MUST be execution-scoped, for example:
	`aecf_prompts/<DOCS_ROOT>/<user_id>/<RUN_DATE>/{{TOPIC}}/executions/{{EXECUTION_ID}}/AECF_00_WORKING_CONTEXT_{{EXECUTION_ID}}.md`
- Reusing WORKING_CONTEXT across different executions is forbidden.

## PHASE 0 — REPOSITORY CONTEXT DISCOVERY (MANDATORY GATE)

### Objective
Collect verified repository context through IDE LLM discovery before any behavioral reasoning.

### Required Action
The executor MUST instruct IDE LLM to perform explicit repository discovery for the requested behavior.

### Mandatory Output
Create `WORKING_CONTEXT` with all fields below.

If any mandatory field is missing, set gate to NO-GO and STOP.

WORKING_CONTEXT lifecycle requirement:
- Primary mode: in-memory within current execution context.
- Optional persisted mode: only with `EXECUTION_ID`-unique artifact name/path.
- Static names like `WORKING_CONTEXT.md` are forbidden.

### ENGINE OUTPUT CONTRACT (CRITICAL — READ BEFORE EXECUTING)

The AECF engine validates the PHASE 0 output text with strict regex rules.
If the output does not match the exact format below, `handle_apply` will
reject it with `Mandatory context contract violation` and the execution
will be **blocked**.

**Rules the engine enforces:**
1. The output text MUST contain `## WORKING_CONTEXT` as a markdown H2 heading.
2. Each subsection MUST use the exact format `### N) SECTION_NAME` (H3 heading with number and closing parenthesis).
3. `SOURCE_REFERENCES` MUST include at least one concrete file path (e.g. `src/app.py:42`).
4. If you delegate discovery to a sub-agent (e.g. `runSubagent`, Plan agent), you MUST reformat the sub-agent results into this exact structure **before** calling apply. Never pass raw sub-agent output directly.

### WORKING_CONTEXT (Required Structure)

The output passed to the engine MUST follow this exact template:

```markdown
## WORKING_CONTEXT

### 1) TARGET_BEHAVIOR
- Observed behavior statement
- Scope boundaries

### 2) ENTRY_POINTS
- Candidate entry files/functions with file references

### 3) EXECUTION_PATHS
- End-to-end path candidates (ordered)

### 4) DATA_INPUTS_AND_STATE
- Inputs, initial state, preconditions

### 5) DECISION_POINTS
- Conditional branches and loop conditions affecting behavior

### 6) DEPENDENCIES
- Internal modules
- External systems (DB/API/queue/filesystem)

### 7) CONFIG_AND_FLAGS
- Environment variables, feature flags, static config impacting behavior

### 8) TEST_AND_LOG_EVIDENCE
- Relevant tests, logs, traces, runtime evidence

### 9) UNCERTAINTIES
- Unknowns, ambiguity, missing evidence

### 10) ASSUMPTIONS
- Explicit assumptions (must be minimal and justified)

### 11) SOURCE_REFERENCES
- File paths and line-level references used in discovery
```

⚠️ DO NOT omit any section. DO NOT rename headings. DO NOT change numbering format.
The engine regex checks for `^## WORKING_CONTEXT` and `^### \d+\) SECTION_NAME` literally.

### PHASE 0 Gate Criteria (mandatory_context)

GO only if:
- All 11 WORKING_CONTEXT fields are present.
- Source references are concrete and traceable.
- Uncertainties and assumptions are explicitly declared.
- Execution context isolation is respected (no shared static file usage).

NO-GO if:
- Any section is missing.
- Discovery includes invented/non-verifiable artifacts.
- References are absent or ambiguous.
- WORKING_CONTEXT is shared across runs or persisted without unique `EXECUTION_ID`.

## PHASE 2 — GOVERNANCE & QUALITY GATES

### Objective
Evaluate analytical quality and classify risks before final decision.

### Quality Dimensions (Mandatory)

1. Code Clarity
- Is the behavior explanation understandable and internally consistent?

2. Coupling
- Are hidden cross-module dependencies influencing the behavior?

3. Testability
- Can the observed behavior be validated/reproduced with concrete checks?

4. Side Effects
- Are side effects explicit and mapped to execution stages?

5. Determinism
- Is the behavior deterministic or conditional on runtime/environment state?

### Risk Classification (Mandatory)

- CRITICAL: High impact, high uncertainty, or governance-breaking condition.
- WARNING: Relevant issue with bounded impact or partial evidence.
- WISH: Improvement opportunity with low immediate risk.

### Gate Criteria (quality_validation)

GO only if:
- All five quality dimensions are assessed.
- At least one reproducibility path is documented.
- Risk matrix is complete and coherent.

NO-GO if:
- Any quality dimension is missing.
- Critical uncertainty blocks causal confidence.
- Risk classification is inconsistent with evidence.

### Output Requirements
Produce `aecf_prompts/<DOCS_ROOT>/<user_id>/<RUN_DATE>/{{TOPIC}}/AECF_02_GOVERNANCE_GATES.md` with:
- Quality gate checklist.
- Risk matrix (CRITICAL/WARNING/WISH).
- Gate result: GO or NO-GO.
